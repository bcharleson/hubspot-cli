# Claude Code Instructions — HubSpot CLI

## Project Overview

JSON-first CLI and MCP server for the HubSpot CRM API v3. Wraps 55 commands across 11 groups into both a terminal CLI and an MCP server for AI agents. Covers contacts, companies, deals, tickets, owners, pipelines, engagements, associations, lists, properties, and universal search.

**Dual interface, single codebase:** Every API endpoint is defined once as a `CommandDefinition` object that powers both the CLI subcommand and the MCP tool.

## Architecture

### CommandDefinition Pattern

Every command is a single `CommandDefinition` object. For standard CRM objects (contacts, companies, deals, tickets), commands are generated by the factory in `src/factories/crm-object.ts`. Custom commands (merge, engagements, associations, search) have hand-written definitions.

```typescript
interface CommandDefinition {
  name: string;           // MCP tool name: "contacts_list"
  group: string;          // CLI group: "contacts"
  subcommand: string;     // CLI subcommand: "list"
  description: string;    // Shared help text
  inputSchema: ZodObject; // Validates CLI flags AND MCP input
  cliMappings: {...};     // Maps Zod fields to Commander args/options
  endpoint: { method, path };
  fieldMappings: {...};   // Where each field goes: path | query | body
  bodyWrapper?: string;   // Optional: wraps body fields under this key (e.g., "properties")
  handler: (input, client) => Promise<unknown>;
}
```

### Key Files

- `src/core/types.ts` — CommandDefinition interface, HubSpotClient interface, config types
- `src/core/client.ts` — HTTP client (Bearer auth, retry on 429/5xx, cursor-based pagination)
- `src/core/handler.ts` — executeCommand() builds HTTP requests from CommandDefinition + input. Supports `bodyWrapper` to nest body fields (e.g., `{ properties: { ...body } }`)
- `src/core/auth.ts` — Access token resolution (--access-token flag > env var > config file)
- `src/core/output.ts` — JSON output formatting, --fields (supports nested paths), --quiet, --pretty
- `src/core/errors.ts` — Typed error classes (AuthError, NotFoundError, ValidationError, RateLimitError, ServerError)
- `src/core/config.ts` — ~/.hubspot-cli/config.json manager
- `src/factories/crm-object.ts` — Factory that generates list/get/create/update/delete/search for any CRM object type
- `src/commands/index.ts` — Command registry (allCommands array), registerAllCommands(), registerCommand()
- `src/mcp/server.ts` — MCP server (registers all CommandDefinitions as tools)
- `src/index.ts` — CLI entry point (Commander)
- `src/mcp.ts` — Direct MCP entry point

## API Specifics

- **Base URL:** `https://api.hubapi.com`
- **Auth:** `Authorization: Bearer <access-token>` (Private App tokens)
- **CRM objects:** All use `/crm/v3/objects/{objectType}` with `{ properties: { ... } }` body wrapper
- **Pagination:** Cursor-based using `after` query param. Response includes `paging.next.after`
- **Associations:** Uses v4 API (`/crm/v4/associations/...`) with batch endpoints
- **Rate limit:** 100 requests per 10 seconds for private apps (handled automatically with retry)
- **Search:** POST to `/crm/v3/objects/{objectType}/search` with `filterGroups`, `query`, `sorts`, `properties`, `limit`, `after`
- **Delete is archive:** DELETE endpoints soft-delete (archive), not hard-delete
- **400 = validation error:** HubSpot returns 400 for invalid property values, bad filters, etc.

## Adding New Commands

### New CRM object type (using factory)

1. Create `src/commands/<objectType>/index.ts`
2. Call `createCrmObjectCommands()` with object config:
   ```typescript
   import { createCrmObjectCommands } from '../../factories/crm-object.js';
   export const allNewObjectCommands = createCrmObjectCommands({
     objectType: 'new_objects',
     singular: 'new object',
     defaultProperties: ['prop1', 'prop2'],
     writeProperties: [
       { field: 'prop1', flags: '--prop1 <value>', description: 'Property 1', required: true },
     ],
   });
   ```
3. Import and spread into `allCommands` in `src/commands/index.ts`
4. Build: `npm run build`

### Custom command

1. Create a `CommandDefinition` object (see existing examples)
2. Use `executeCommand()` for standard REST endpoints, or write a custom handler
3. Add to the appropriate group's exports
4. Register in `allCommands`

### Adding `bodyWrapper`

Set `bodyWrapper: 'properties'` on CommandDefinition to automatically wrap body fields:
```typescript
// Input: { email: "john@acme.com", firstname: "John" }
// With bodyWrapper: 'properties', sends: { properties: { email: "john@acme.com", firstname: "John" } }
```

## Important Conventions

- **@inquirer/prompts is external** — marked as external in tsup.config.ts and dynamically imported only in login.ts. This ensures Node 18 compatibility for all non-interactive commands.
- **Zod validation runs before API calls** — `registerCommand()` validates input against the schema and gives clear "Missing required option(s)" errors.
- **Unknown subcommand errors list available commands** — each group has a `command:*` handler.
- **All output is JSON to stdout** — never use console.log for anything except structured output. Use console.error for errors.
- **MCP tools share the same handler** — no divergence between CLI and MCP behavior.
- **Factory-generated commands** cover contacts, companies, deals, and tickets. Each gets list/get/create/update/delete/search automatically.

## Do Not

- Do not add static imports of `@inquirer/prompts` — always use dynamic `await import('@inquirer/prompts')`
- Do not modify the output format — agents depend on compact JSON to stdout
- Do not add interactive prompts to API commands — only login should be interactive
- Do not create new command files without adding them to `allCommands` in `src/commands/index.ts`
- Do not bypass the `bodyWrapper` pattern — HubSpot CRM objects always expect `{ properties: { ... } }`
- Do not hardcode property names in factory commands — use the `defaultProperties` and `writeProperties` config
